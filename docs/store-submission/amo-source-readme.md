# AMO source-submission README (DocRewind, Firefox)

For addons.mozilla.org (AMO) reviewers. AMO requires that a reviewer can rebuild
the extension from source and confirm the submitted artifact matches. DocRewind's
build is **deterministic from the committed lockfile (same-environment)** — see
"Reproducibility scope" below for exactly what that claim covers.

## What is submitted

- The Firefox extension archive: `docrewind-<version>-firefox.zip`.
- The source archive WXT emits alongside it: `docrewind-<version>-sources.zip`
  (produced automatically by `bun run zip:firefox`; contains the full first-party
  source — `wxt.config.ts`, `uno.config.ts`, `entrypoints/`, `components/`,
  `lib/`, `package.json`, `bun.lock`, etc., all human-readable).

## Toolchain (exact)

- **Bun 1.3.14** (exact pin — see `docs/STACK.md` and `.augment/rules/bun-solid-pro.md`).
- **WXT 0.20.26** (in `package.json`); the manifest is generated by WXT, never
  hand-written (`wxt.config.ts` is the single source of permission/identity facts).
- No Node toolchain is required separately; Bun runs the build.

## Rebuild from source

From the unpacked `-sources.zip` (or a clone at the tagged commit):

```bash
bun install --frozen-lockfile     # installs the EXACT pinned graph from bun.lock
bun run postinstall               # wxt prepare
bun run build:firefox             # → .output/firefox-mv3/
bun run zip:firefox               # → .output/docrewind-<version>-firefox.zip
```

To confirm the build is well-formed and carries the minimal permission set:

```bash
bash scripts/verify-manifest.sh   # asserts permissions==[storage],
                                  # host_permissions==[*://docs.google.com/*],
                                  # no <all_urls>, no remote code, gecko.id + event-page
```

## Diff against the submitted artifact

Extract both the rebuilt zip and the submitted zip and compare their **contents**
(zip container metadata — entry order, mtimes — is environment-dependent and not
meaningful; compare the extracted files):

```bash
mkdir built submitted
unzip -q .output/docrewind-<version>-firefox.zip -d built
unzip -q /path/to/submitted-docrewind-<version>-firefox.zip -d submitted
diff -r built submitted        # expect: no differences in file contents
```

The repository also ships `scripts/verify-reproducible-build.sh`, which builds
twice in one environment and asserts the two builds are per-file content-identical
(the same property a reviewer's rebuild relies on).

## Reproducibility scope (claim only what is proven)

DocRewind claims **determinism from the committed lockfile, same-environment**: on
the pinned Bun 1.3.14 toolchain, a rebuild produces per-file content-identical
extension archives. This is the property AMO source review actually needs — a
reviewer on a comparable environment can rebuild and diff to confirm the artifact.

**Cross-machine byte-for-byte reproducibility is a PRD §0.9 stretch goal, not a
shipped guarantee** (a pinned-container recipe is sketched in `docs/RELEASE.md`
but not built). The wording here is kept identical in scope to `docs/STACK.md` and
`docs/RELEASE.md`.

## Minified, not obfuscated

The build minifies with esbuild (Vite default) **with sources retained**.
There is **no obfuscation** (no `javascript-obfuscator`/terser-obfuscate plugin in
`wxt.config.ts`); `verify-manifest.sh` asserts both the absence of an obfuscation
plugin and that `-sources.zip` carries readable source. This satisfies AMO's
"reviewable source" requirement (PRD §11.4).

## Privacy posture (for the reviewer)

DocRewind makes network requests to `docs.google.com` only, processes everything
locally, and transmits no user data. This is enforced at runtime
(`e2e/network-isolation.spec.ts`) and by static lint
(`scripts/check-no-foreign-hosts.sh`).

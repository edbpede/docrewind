# Firefox manual validation (Phase 6 / PRD §9.10, §11.5)

DocRewind ships cross-browser (Chrome MV3 + Firefox MV3), but automated E2E is
**Chromium-only** — Playwright can only load an extension into a persistent
Chromium context. Firefox therefore has **no automated E2E**; its coverage is a
manual exploratory pass plus a `web-ext` smoke run, executed **once per release**
against a real `build:firefox` artifact.

> **Status:** This checklist must be executed at least once against a real
> `bun run build:firefox` artifact before a release that targets Firefox. As of
> Phase 6 it is authored but **not yet executed** — the run is a release gate,
> not a CI gate (CI provisions Chromium only).

## Build the Firefox artifact

```bash
bun install --frozen-lockfile
bun run postinstall          # wxt prepare
bun run build:firefox        # → .output/firefox-mv3/
```

The Firefox manifest is generated by WXT from `wxt.config.ts`
(`browser_specific_settings.gecko.id = docrewind@edbpede.github.io`;
`host_permissions = ["*://docs.google.com/*"]`; `permissions = ["storage"]`).
Firefox treats `host_permissions` as opt-in, so the first activation on
`docs.google.com` may prompt for the per-site grant — accept it.

## `web-ext` smoke run

```bash
# One-time: bun add -g web-ext   (or use bunx web-ext@latest)
bunx web-ext run -t firefox-desktop --source-dir .output/firefox-mv3
```

Smoke sequence (the happy path must work end to end):

1. **Load** — Firefox launches with the temporary add-on installed; no manifest
   or console errors on startup.
2. **Activate** — open a Google Doc you own at `https://docs.google.com/document/d/<id>/edit`;
   trigger DocRewind (toolbar action) and confirm it activates only on
   `docs.google.com` (no activation on unrelated origins).
3. **Retrieve** — start retrieval; confirm the revision range is discovered and
   chunks are fetched (progress advances; no non-Google network requests — verify
   in the Network panel).
4. **Replay** — open the replay surface (`replay.html?doc=<id>`); confirm the
   document reconstructs, Play/Pause works, and the timeline scrubs.

## Manual exploratory matrix

Run each row against a real doc and record pass/fail + notes. Cover Firefox-
specific rendering and the document shapes most likely to stress decode/
reconstruction:

| # | Document shape | What to verify |
|---|----------------|----------------|
| 1 | **Small doc** (a few revisions) | Full reconstruction; end-of-timeline text == current text. |
| 2 | **Long doc** (hundreds of revisions) | Chunked retrieval resumes/completes; replay stays responsive; memory stable. |
| 3 | **Collaborative doc** (multiple authors) | Per-session grouping in the timeline; author attribution preserved. |
| 4 | **Paste-heavy doc** | Large-insertion events detected; pasted blocks render in order. |
| 5 | **Comments / suggestions** | Suggestion ops decode without aborting; comments do not corrupt the text stream. |
| 6 | **Images / tables** | Non-text ops isolated as opaque placeholders; position + timing preserved; no decode abort. |
| 7 | **Mixed-language / RTL** | Unicode + bidi text reconstructs correctly; scrubbing preserves grapheme boundaries. |
| 8 | **Multi-account session** (`/document/u/{N}/d/`) | Correct user-index slot detected; retrieval succeeds on the non-zero account. |

## Privacy spot-check (manual)

While exercising the matrix, keep the Network panel open and confirm **zero
requests leave `docs.google.com`** (plus first-party `moz-extension://`
housekeeping). This mirrors the automated Chromium network-isolation audit
(`e2e/network-isolation.spec.ts`) for the Firefox runtime, which that audit
cannot reach.
